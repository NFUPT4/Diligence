# 贡献指南

感谢你参与本项目！为了方便协作、保证代码质量，请仔细阅读以下内容，并遵循项目和团队的开发流程。

---

## 1. 任务分配与看板

- 项目管理员会将任务直接分配给对应的成员，所有任务都可以在 **Issues** 和 **项目看板（Project Board）** 中查看。  
- 每个任务通常包含 **优先级（Priority）**、**标签（Labels）** 等必要信息，帮助你快速了解需要做什么。  
- **主动领取任务**：如果某个 Issue 没有指派（Assignee 为空），且你有意愿和能力完成，可以直接在该 Issue 下留言告知管理员，或自行将自己设为 Assignee，即可开始工作。  
- 任何成员都可以根据实际需要 **自行创建 Issue**，例如提出新功能、报告缺陷、改进文档等。

> 示例：当你发现一个未被认领的 Issue，如 `#42 添加用户头像上传功能`，你可以在评论区中 @ 管理员说“我来做这个”，然后将 Issue 指派给自己。

---

## 2. 分支命名规范

为避免混乱，每位成员的工作都应该在 **自己的分支** 上进行，并且按照工作类型使用不同的分支前缀。

| 分支类型 | 命名格式 | 用途说明 |
|---------|----------|----------|
| 功能开发 | `feat/xxx` | 新功能、新特性 |
| 紧急修复 | `hotfix/xxx` | 线上版本或主分支的紧急故障修复 |
| 文档修改 | `docs/xxx` | 仅修改文档、注释等 |
| 测试相关 | `test/xxx` | 添加或修改测试代码 |
| 构建部署 | `build/xxx` | 构建脚本、CI/CD、打包等相关改动 |
| 日常维护 | `chore/xxx` | 依赖更新、代码清理、配置文件调整等杂项 |

- 示例：`feat/user-avatar-upload`、`hotfix/login-crash`、`docs/api-readme`、`test/user-module`、`build/add-docker`、`chore/upgrade-libs`。  
- 分支名建议使用 **英文小写 + 短横线** 连接，简短且可读。

## 3. 日常开发流程

### 3.1 保持代码最新

在开始工作之前，请务必 **频繁从主分支拉取最新代码**，避免后期出现过多冲突。

```bash
# 切换到主分支（假设为 main）
git checkout main
git pull origin main

# 再基于最新的 main 创建你的工作分支
git checkout -b feat/some-feature
```

### 3.2 Commit 消息规范

建议每次提交时，在消息中简明地说明做了什么，并且关联对应的 Issue。

- **修复类提交**  
  `fix: #123 修复用户头像上传失败的问题`
- **功能类提交**  
  `feat: #456 添加裁剪头像的功能`
- **文档、杂项等**  
  `docs: #78 补充接口文档示例`  
  `chore: 升级 axios 至 1.x`

> 更规范的格式可参考：`<type>: #<issue_id> <简短描述>`，其中 `<type>` 可以是 `feat`、`fix`、`docs`、`test`、`build`、`chore` 等。

### 3.3 发起 Pull Request

在你的分支开发完成且本地测试通过后，向管理员发起 **Pull Request（PR）**。

PR 的 **描述（Description）** 末尾，必须按照以下格式引用或关闭对应的 Issue：

| 情形 | 写法示例 | 效果 |
|------|---------|------|
| 修复了问题，需要关闭 Issue | `Fixes #123` | PR 合并后自动关闭 #123 |
| 新增了功能，需要关闭 Issue | `Closes #456` | PR 合并后自动关闭 #456 |
| 纯粹引用 Issue，不需要关闭 | `Refs #678` | 仅建立关联，不自动关闭 |

- 关于 Issue 编号：  
  每个 Issue 都有一个唯一引用编号，一般为 `#123` 这样的格式，也可能是 `#IJEBM6` 这样的短 ID（视具体项目工具而定）。你可以在 **Issue 标题下方的信息栏** 中找到它。

> PR 描述示例：
> ```
> ## 变更说明
> 为头像上传增加了裁剪功能，用户可以在选择图片后调整尺寸。
> 
> Closes #42
> ```

### 3.4 Code Review 与合并

- 管理员会对你的 PR 进行 **代码审查（Code Review）**，并在评论区提出修改建议。
- 如果审查通过，管理员会直接将你的分支 **合并到主分支**；若存在问题，你需要根据反馈继续修改，并推送新的提交到同一分支，PR 会自动更新，直至达到合并标准。
- 合并完成后，管理员会 **关闭对应的 Issue**，同时在项目看板中更新任务状态为“已完成”或类似列。

## 4. 测试人员 Issue 规范

负责测试的成员在测试过程中发现缺陷或体验问题时，应当 **新建 Issue** 来记录，并提供以下信息：

- **问题描述**：具体操作步骤、预期行为和实际行为。
- **环境信息**：操作系统、浏览器版本、测试环境等。
- **附件/截图**：如有必要，附上截图或日志。
- **严重程度**：可配合使用标签，如 `bug`、`critical`、`enhancement` 等。

> bug issue 示例:
>
> ```markdown
> # 问题描述
> <!-- 用一两句话清晰描述遇到了什么问题 -->
> 
> ## 复现步骤
> <!-- 列出能够稳定复现该问题的操作步骤，最好每一步都编号 -->
> 1. 打开 '...'
> 2. 点击 '...'
> 3. 滚动到 '...'
> 4. 看到错误
> 
> ## 期望结果
> <!-- 你期望在这一步看到什么现象或得到什么结果 -->
> 
> ## 实际结果
> <!-- 实际发生了什么，包括错误信息、界面异常等 -->
> 
> ## 截图或录屏
> <!-- 如果有截图、GIF 或录屏，请直接粘贴或拖拽到此处 -->
> 
> ## 错误日志
> <!-- 如果有控制台错误、崩溃日志或堆栈信息，请提供关键片段 -->
> 
> ## 环境信息
> - 操作系统及版本：（例如 Windows 11 / macOS 14 / Ubuntu 22.04）
> - 软件/应用版本：（例如 v2.3.1 或 commit 号）
> - 浏览器及版本（如适用）：（例如 Chrome 120）
> - 设备型号（如适用）：（例如 iPhone 15 Pro）
> - 其他可提供的信息：
> 
> ## 补充说明
> <!-- 任何其他你认为有助于定位问题的信息，比如是否仅在特定网络环境出现、是否尝试过重装等 -->
> ```
> 
> **使用建议：**
> 1. **标题**请改成一句能概括问题的简短描述，如：`点击保存按钮后页面无响应`。
> 2. 将 `<!-- ... -->` 内的注释提示删掉，或根据提示填写内容。
> 3. 尽量附上截图或日志，能极大提高被处理的速度。

## 5. 冲突解决指南

当你 `git pull` 或合并分支时，可能会遇到类似以下标记的 **冲突（Conflicts）**：

```
<<<<<<< HEAD
当前分支的代码
=======
你拉取下来的代码
>>>>>>> branch-name
```

解决步骤：

1. 仔细判断哪部分代码应该保留，或手动整合两边的修改。
2. 删除冲突标记 `<<<<<<<`、`=======` 和 `>>>>>>>`，只保留最终想要的代码。
3. 保存文件后，执行以下命令完成解决：
   ```bash
   git add <冲突文件>
   git commit -m "解决合并冲突 - <简要描述>"
   ```
4. 如果遇到复杂的冲突，建议与相关代码的作者沟通确认后再操作，避免误删有效代码。
